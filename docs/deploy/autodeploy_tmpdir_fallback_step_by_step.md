# Автодеплой без падений при переполнении `/tmp` (шаг за шагом)

## Когда нужен этот сценарий
Если в CI видно ошибку вида `mktemp: ... No space left on device`, значит временный каталог по умолчанию на сервере переполнен.

## Что уже реализовано в проекте
- `scripts/deploy.sh` теперь использует `DEPLOY_TMPDIR` и fallback-логику для `mktemp`.
- Если системный `/tmp` недоступен, временные файлы создаются в каталоге проекта (`$DEPLOY_PATH/.tmp` по умолчанию).

## Как включить в GitHub Actions
1. Откройте `Settings -> Secrets and variables -> Actions`.
2. Создайте секрет `DEPLOY_TMPDIR` (рекомендуемое значение: `/var/www/numerolog_bot/.tmp`).
3. Убедитесь, что пользователь деплоя имеет права на запись в эту директорию.
4. Запустите workflow `Landing CI/CD (VPS + Nginx)` вручную через `workflow_dispatch`.

## Проверка после деплоя
1. В логах шага `Deploy (git reset + restart service + smoke-check)` убедитесь, что деплой завершился строкой:
   - `[OK] Деплой успешно завершен. Маркер: .../.last_deploy_success`
2. На сервере проверьте, что каталог существует:
   - `ls -la /var/www/numerolog_bot/.tmp`
3. Если места всё ещё мало — очистите старые временные файлы и логи:
   - `find /var/www/numerolog_bot/.tmp -type f -mtime +7 -delete`

## Рекомендации по надёжности
- Добавьте cron-задачу на очистку временных файлов старше 7 дней.
- Настройте мониторинг свободного места (`df -h`) и алерты при заполнении > 85%.
- Держите `DEPLOY_TMPDIR` на том же диске, где расположен `DEPLOY_PATH`, чтобы fallback был предсказуемым.
